算法 —— 动态规划

问题说明

动态规划问题，
一般求解思路：

  1. 【最优子结构】找到最优子结构(子问题)，如F(10)可以分成F(9),F(8)两部分。
  2. 【状态转移方程】描述状态转移方程(用子问题的解来表示原问题），如F(N) = F(N-1)+F(N-2)
    问题的边界,如，F(1)=1,F(2)=2.
  3. 最终实现一般有：传统递归方法、备忘录法、动态规划法。

明确 base case -> 明确「状态」-> 明确「选择」 -> 定义 dp 数组/函数的含义。

路由网关 —— Zuul

  微服务网关涉及到负载均衡，缓存，路由，访问控制，服务代理，监控，日志等功能，

1. zuul的功能

  zuul的主要功能如下：

  • 动态路由和负载均衡；
  • 统一api的暴露，屏蔽内部服务敏感信息；
  • 身份认证和权限认证，安全保护；
  • 监控及日志输出，记录请求；
  • 流量监控，服务降级；
  • api接口从内部服务分离，方便测试。

2. zuul的工作原理

  zuul通过自定义的ZuulServlet对请求进行控制，通过一系列的过滤器，在http请求的发起和响应期间进行一系列的操作。zuul采用动态读取、编译和运行这些过滤器，这些过滤器之间不能互相直接通信，而是使用RequestContext对象来共享数据，每个请求都会创建一个RequestContext。
  zuul在2.0版本对于现有框架进行了升级，使用了netty的nio机制提升性能，但是官方给出的性能提升数据大致在20%，所以如果不是异步场景较多，仍然考虑采用1.0版本，本文仍然以1.0版本为例。

  外部http请求到达api网关服务的时候，首先它会进入第一个阶段pre，在这里它会被pre类型的过滤器进行处理。该类型过滤器的主要目的是在进行请求路由之前做一些前置加工，比如请求的校验等。
  在完成了pre类型的过滤器处理之后，请求进入第二个阶段routing，也就是之前说的路由请求转发阶段，请求将会被routing类型的处理器处理。这里的具体处理内容就是将外部请求转发到具体服务实例上去的过程,
  当服务实例请求结果都返回之后，routing阶段完成，请求进入第三个阶段post。此时请求将会被post类型的过滤器处理，这些过滤器在处理的时候不仅可以获取到请求信息，还能获取到服务实例的返回信息，所以在post类型的过滤器中，我们可以对处理结果进行一些加工或转换等内容。
  还有一个特殊的阶段error，该阶段只有在上述三个阶段中发生异常的时候才会触发，但是它的最后流向还是post类型的过滤器，因为它需要通过post过滤器将最终结果返回给请求客户端（对于error过滤器的处理，在spring cloud zuul的过滤链中实际上有一些不同）

  zuul2.0前端用Netty Server代替Servlet，目的是支持前端异步。后端用Netty Client代替Http Client，目的是支持后端异步，过滤器换了名字，用Inbound Filters代替Pre-routing Filters，用Endpoint Filter代替Routing Filter，用Outbound Filters代替Post-routing Filters。

4. 原理剖析

4.1 过滤器

  通过shouldFilter()指定是否开启过滤器，如果是run()方法中的过滤器逻辑才会生效；通过filterType()指定过滤器类型；通过filterOrder()指定过滤顺序，值越小，优先级越高；通过重写run()来实现真实的过滤逻辑。
filterType

  • pre过滤器
      可以进行一些权限认证，日志记录，或者额外给Request增加一些属性供后续的filter使用。pre会优先按照order从小到大执行，然后再去执行请求转发到业务服务；
  • routing过滤器
  • post过滤器
      在post的filter里，一般做一些日志记录，或者额外增加response属性；
  • error过滤器
      对于同一种类型的过滤器，可以通过设置不同的优先级来定义生效的时机，比如说在pre过滤器中可以有两个自定义的过滤器，只需要设置filterOrder为0和1即可控制生效的先后顺序。

多线程编程（2） —— synchronized关键字

  “非线程安全”指的是在多个线程对同一个对象中的实例变量进行并发访问时发生，产生的结果是“脏读”，即渠道的数据其实被更改过，“线程安全”指的是获得的实例变量的值是经过同步处理的，不会出现脏读问题。

1.1 方法内的变量为线程安全

  如果变量是在方法内的，那么不存在“非线程安全”问题。

多线程编程

多线程编程（1） —— 多线程开启方式

  1个进程至少包含1个线程，多线程的使用方式有两种：继承Thread类，实现Runnable接口。

1. 继承Thread类

  Thread类本身也实现了Runnable接口，但是如果需要创建线程的类已经有了父类，就无法使用这种方式，因为java不支持多继承。

1.1 线程调用的随机性

  从运行结果来看，MyThread.java类中的run方法执行的时间比较晚，这也说明在使用多线程技术时，代码的运行结果与代码执行顺序或者调用顺序是无关的。线程是一个子任务，CPU以不确定的方式，或者说以随机的时间来调用线程中的run方法，所以可能出现先打印“运行结束！”后输出“MyThread”这样的结果。

1.2 线程启动的随机性

  使用随机时间的sleep，使得线程挂起，此时CPU会执行不同的线程。
  Thread的start()方法通知“线程规划器”，此线程已经准备就绪，等待调用线程对象的run()方法，即就绪之后，等待一段时间，使得系统能够安排调用Thread的run()方法，使得线程真正开始运行，启动线程，相对于“异步执行”。如果直接调用thread.run()则为“同步执行”，整个执行动作直接由线程发起，而不是等待系统发起调用。
要点：执行start()只代表准备就绪。

  这里如果不添加Thread.sleep(2），如果cpu切换的足够快，就无法观察到。

  显式的使得线程sleep时间为2ms，则可以看到执行start()方法的顺序不代表线程启动的顺序。

2. 实现Runnable接口

  使用继承Thread类的方式来开发多线程的设计有局限性，由于java对于多继承不支持，所以使用实现Runnable接口的方式来完成，

  将Runnable接口的对象传入到Thread的构造函数的参数中，调用thread的start()方法，该线程的处理逻辑在run()方法中描述。
