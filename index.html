java值传递和引用传递

在Java方法中参数列表有两种类型的参数，基本类型和引用类型。

基本类型：byte,short,int,long,char,float,double,Boolean,returnAddress，
值存放在局部变量表中，无论如何修改只会修改当前栈帧的值，方法执行结束对方法外不会做任何改变；此时需要改变外层的变量，必须返回主动赋值。
方法调用时，实际参数把它的值传递给对应的形式参数，函数接收的是原始值的一个copy，此时内存中存在两个相等的基本类型，即实际参数和形式参数，后面方法中的操作都是对形参这个值的修改，不影响实际参数的值。

引用数据类型：类类型，接口类型和数组
指针存放在局部变量表中，调用方法的时候，副本引用压栈，赋值仅改变副本的引用。但是如果通过操作副本引用的值，修改了引用地址的对象，此时方法以外的引用此地址对象当然被修改。（两个引用，同一个地址，任何修改行为2个引用同时生效）。
方法调用时，实际参数的引用(地址，而不是参数的值)被传递给方法中相对应的形式参数，函数接收的是原始值的内存地址；
在方法执行中，形参和实参内容相同，指向同一块内存地址，方法执行中对引用的操作将会影响到实际对象。

这两种类型都是将外面的参数变量拷贝一份到局部变量中，基本类型为值拷贝，引用类型就是将引用地址拷贝一份。

线程通知与等待

线程通知与等待

1. wait()函数

当一个线程调用一个共享变量的wait()方法时，该调用线程会被阻塞挂起，直到发生以下之一才会返回：

  • 其他线程调用了该共享对象的notify()或notifyAll()方法；
  • 其他线程调用了该线程的interrupt()方法，该线程抛出InterruptedException异常返回；

如果调用wait()方法的线程没有事先获取该对象的监视器锁，则调用wait()方法时调用线程会抛出IlegealMonitorStatException异常。


线程如何获取到共享变量的监视器锁？

  • 执行synchronized同步代码块时，使用该共享变量作为参数。

  • 调用该共享变量的方法，并且该方法使用了synchronized修饰。


一个线程可以从挂起状态变为可运行状态（也就是被唤醒），即便该线程没有被其他线程调用notify()、notifyAll()方法进行通知，或者被中断，或者等待超时，这就是 虚假唤醒。


由于虚假幻想存在的可能性，需要不停的测试该线程被唤醒的条件是否满足，不满足则继续等待。

通过同步块获取obj的监视器锁，然后再while循环中调用obj的wait()方法。

算法 —— 动态规划

问题说明

  动态规划在寻找有很多重叠子问题的情况下的最优解时有效，将大问题分解为子问题，最核心的部分是利用历史记录来保存中间子问题的答案，避免重复计算，这些历史记录需要进行存储，需要借助于一定的数据结构，如一维数字或者二维数组等。

  如果一个问题满足以下2点，那么它就能用动态规划解决。

  1. 问题的所有答案构成一个潜在的数列
  2. 大问题最终可以由小问题来递推得到，f(n) 可以从f(n-1)获取到。

  当然，上面的提炼，除了把新手直接吓懵，没有任何作用，所以我们从基础入手，逐步揭开 “动态规划”的神秘面纱。

举个例子

跳台阶问题

求解思路

  1. 【最优子结构】找到最优子结构(子问题)，如F(10)可以分成F(9),F(8)两部分。
  2. 【状态转移方程】描述状态转移方程(用子问题的解来表示原问题），如F(N) = F(N-1)+F(N-2)
    问题的边界,如，F(1)=1,F(2)=2.
  3. 最终实现一般有：传统递归方法、备忘录法、动态规划法。

明确 base case -> 明确「状态」-> 明确「选择」 -> 定义 dp 数组/函数的含义。

路由网关 —— Zuul

  微服务网关涉及到负载均衡，缓存，路由，访问控制，服务代理，监控，日志等功能，

1. zuul的功能

  zuul的主要功能如下：

  • 动态路由和负载均衡；
  • 统一api的暴露，屏蔽内部服务敏感信息；
  • 身份认证和权限认证，安全保护；
  • 监控及日志输出，记录请求；
  • 流量监控，服务降级；
  • api接口从内部服务分离，方便测试。

2. zuul的工作原理

  zuul通过自定义的ZuulServlet对请求进行控制，通过一系列的过滤器，在http请求的发起和响应期间进行一系列的操作。zuul采用动态读取、编译和运行这些过滤器，这些过滤器之间不能互相直接通信，而是使用RequestContext对象来共享数据，每个请求都会创建一个RequestContext。
  zuul在2.0版本对于现有框架进行了升级，使用了netty的nio机制提升性能，但是官方给出的性能提升数据大致在20%，所以如果不是异步场景较多，仍然考虑采用1.0版本，本文仍然以1.0版本为例。

  外部http请求到达api网关服务的时候，首先它会进入第一个阶段pre，在这里它会被pre类型的过滤器进行处理。该类型过滤器的主要目的是在进行请求路由之前做一些前置加工，比如请求的校验等。
  在完成了pre类型的过滤器处理之后，请求进入第二个阶段routing，也就是之前说的路由请求转发阶段，请求将会被routing类型的处理器处理。这里的具体处理内容就是将外部请求转发到具体服务实例上去的过程,
  当服务实例请求结果都返回之后，routing阶段完成，请求进入第三个阶段post。此时请求将会被post类型的过滤器处理，这些过滤器在处理的时候不仅可以获取到请求信息，还能获取到服务实例的返回信息，所以在post类型的过滤器中，我们可以对处理结果进行一些加工或转换等内容。
  还有一个特殊的阶段error，该阶段只有在上述三个阶段中发生异常的时候才会触发，但是它的最后流向还是post类型的过滤器，因为它需要通过post过滤器将最终结果返回给请求客户端（对于error过滤器的处理，在spring cloud zuul的过滤链中实际上有一些不同）

  zuul2.0前端用Netty Server代替Servlet，目的是支持前端异步。后端用Netty Client代替Http Client，目的是支持后端异步，过滤器换了名字，用Inbound Filters代替Pre-routing Filters，用Endpoint Filter代替Routing Filter，用Outbound Filters代替Post-routing Filters。

4. 原理剖析

4.1 过滤器

  通过shouldFilter()指定是否开启过滤器，如果是run()方法中的过滤器逻辑才会生效；通过filterType()指定过滤器类型；通过filterOrder()指定过滤顺序，值越小，优先级越高；通过重写run()来实现真实的过滤逻辑。
filterType

  • pre过滤器
      可以进行一些权限认证，日志记录，或者额外给Request增加一些属性供后续的filter使用。pre会优先按照order从小到大执行，然后再去执行请求转发到业务服务；
  • routing过滤器
  • post过滤器
      在post的filter里，一般做一些日志记录，或者额外增加response属性；
  • error过滤器
      对于同一种类型的过滤器，可以通过设置不同的优先级来定义生效的时机，比如说在pre过滤器中可以有两个自定义的过滤器，只需要设置filterOrder为0和1即可控制生效的先后顺序。
